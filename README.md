1)Kiedy Git robi fast-forward, a kiedy merge commit? Zauważyłem to przy zadaniach 4 i 5. Fast-forward jest wtedy, gdy w main nic się nie zmieniło i Git po prostu „przepina” historię dalej. Z kolei merge commit powstaje, gdy pracowałem na dwóch gałęziach naraz i w obu są nowe zmiany – wtedy Git musi stworzyć dodatkowy punkt, żeby to wszystko spiąć w całość.
2)Czym w praktyce różni się merge od rebase? Dla mnie merge to bezpieczne łączenie, gdzie widać każde rozgałęzienie i historię taką, jaka była naprawdę. Rebase to taki sposób na „wyprostowanie” historii – moje zmiany z nowej gałęzi trafiają na samą górę maina, przez co wszystko wygląda, jakby było pisane jedno po drugim, bez żadnych bocznych gałęzi. 
3)Jak rozwiązałem konflikt w moim repozytorium? Specjalnie zmieniłem tę samą linię kodu w dwóch gałęziach. Gdy Git wywalił błąd przy łączeniu, otworzyłem plik, ręcznie usunąłem te dziwne znaczki od Gita i zostawiłem tylko ten kod, o który mi chodziło. Na koniec zrobiłem zwykły commit i sprawdziłem, czy program dalej działa.
